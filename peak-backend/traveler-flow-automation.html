0 Full story in order

Below is the real sequence of events when someone triggers a Travelers quote. Read top to bottom: each step names who acts and what happens before the next step can occur.

  1. User or frontend asks for a quote

    The app sends a request to peak-backend to start automation. The body usually includes review data (payload) and ideally a stable extraction id so double-clicks do not start two Skyvern runs for the same document. The backend answers quickly with “accepted” and hands back identifiers the rest of the system will use.

  2. Backend creates a fresh job identity

    For each new run the backend generates a job id (and usually a quote id). This job id is known before Skyvern returns its own run id. That is important: we can build OTP URLs and tell n8n which job this OTP belongs to without waiting for the browser to start.

  3. Backend asks Skyvern to start the workflow

    The integration calls the Skyvern API with the workflow id and parameters (Travelers carrier, credentials reference, payload, job id, quote id). Skyvern schedules a run and eventually opens a real browser session against the Travelers agent site.

  4. Skyvern responds with a workflow run id

    When the API returns, the backend reads Skyvern’s run id (the wr_… value). It stores a mapping in Redis: this job id goes with this Skyvern run id. That mapping expires after a few minutes, same idea as OTP freshness.

  5. Skyvern drives login in the browser

    The agent fills username and password from saved credentials, submits the form, and may land on an identity check screen (for example “send SMS code” and “enter code”). The login block may also be configured with a verification URL Skyvern can call to obtain a code while the session waits.

  6. Travelers sends an OTP (typically SMS)

    The carrier does not call our API directly. The code appears on a phone or, if the team set it up, arrives as email after SMS-to-email or forwarding into the automation mailbox (for example automation@…).

  7. n8n notices new mail in the monitored inbox

    The Outlook (or IMAP) trigger runs when a new message arrives. A code node reads subject and body, finds a six-digit code, and prepares a small JSON object: carrier, otp, timestamps, and correlation fields (job id and/or Skyvern run id when available).

  8. n8n calls the backend OTP webhook

    An HTTP request posts that JSON to peak-backend. If a shared secret is configured, the request must include it or the backend rejects the call. On success, the backend stores the OTP in Redis under a key tied to carrier and Skyvern run id (and keeps the latest job mapping separate for lookups by job id).

  9. Skyvern polls the backend for the code

    While the MFA screen is open, Skyvern repeatedly calls the public OTP URL you configured (often via ngrok), passing the job id as a query parameter. The backend resolves job id → Skyvern run id, then reads the OTP record from Redis. When a code is present, the response body is plain six digits so the automation client can paste it into the field. Polling may happen more than once; the backend is designed so Skyvern can retry until verify succeeds or times out.

  10. Agent enters the code and continues

    If the code is correct and the page accepts it, the session moves past MFA toward dashboard or quote flow. If the code is wrong, expired, or the UI never receives a successful poll, the run may terminate even though one HTTP request returned success earlier.

  11. Parallel runs and stale job ids

    If a new quote is started while an old run still polls, the “latest job” pointer updates. Polls that still use an old job id may be rejected on purpose so two automations do not steal each other’s OTP. Operational fix: cancel old Skyvern runs and avoid double-submitting generate-quote; sending the same extraction id helps the backend dedupe within the TTL window.

Client / IT alignment — Real Travelers OTPs must reach the mailbox (or path) n8n reads. Twilio or SMS-to-email forwarding are org decisions; this document describes how pieces connect once OTP data is available to automation.

1 Who does what (roles)

Think of four cooperating systems. None of them alone completes the quote; they pass responsibility forward in time.

System Responsibility in this flow
peak-backend Accepts quote requests, starts Skyvern, stores job↔run mapping and OTP in Redis, exposes webhooks and the URL Skyvern polls for codes.
Skyvern Runs the browser, logs into Travelers, handles MFA by fetching a code from your backend when configured, then continues the scripted workflow.
n8n Watches email, extracts the six-digit OTP, and pushes it securely into peak-backend so Redis holds the code Skyvern will read.
Redis Short-lived memory shared by backend processes: mappings and OTP records with TTL so old runs do not linger forever.

Order reminder: generate-quote and Skyvern start happen before the OTP is in Redis. The OTP path (carrier → inbox → n8n → webhook) can finish slightly before, during, or after Skyvern hits the MFA screen; polling bridges the timing gap.

2 Redis — why it exists in this flow

Redis is a small, fast database kept in memory. We use it as a shared scratchpad between HTTP requests: one request stores an OTP, another request (from Skyvern) reads it seconds later. Nothing permanent lives there—entries expire automatically.

  1. What we store conceptually

    (a) Job ↔ Skyvern run — after the backend starts Skyvern, it remembers which workflow run id belongs to which job id until TTL expires. (b) OTP by carrier + run id — when n8n posts a code, we file it under the Travelers run so Skyvern’s poll can find it. (c) Latest job pointer — optional guard so stale polls from an older run do not hijack the newest run. (d) In-flight quote dedupe — optional slot so the same extraction id does not spawn two Skyvern runs within the TTL window.

  2. Why TTL matters

    OTPs are only valid for a short time on the carrier side. Automatically deleting Redis keys after a few minutes avoids serving ancient codes and reduces risk if someone ever read stale data.

  3. Operator setup (short checklist)

    Run a Redis instance reachable from peak-backend (commonly Docker on port 6379). Point the backend at it via environment configuration. Use the project’s Redis smoke script once to confirm connectivity before relying on OTP flows.

3 Backend — quote start, OTP ingest, Skyvern poll

peak-backend is the hub. It does not read email itself; it exposes HTTP endpoints that the frontend, n8n, and Skyvern each call at different moments.

  1. Generate quote (first external call in the automation)

    The client calls the backend to start a Travelers run. The backend creates identifiers, calls Skyvern’s API, then saves job↔run mapping when Skyvern answers. The HTTP response returns those identifiers to the client so tooling (and n8n bridges) can correlate OTPs with the right browser session.

  2. OTP webhook (n8n → backend, after mail arrives)

    This is the second major path. n8n sends the six-digit code plus correlation: the Skyvern workflow run id (and optionally the same job id the backend already issued). The backend checks a shared secret if configured, validates the code shape, resolves the real Skyvern run id if n8n only had a job id, then writes the OTP into Redis against that run.

  3. Skyvern OTP poll URL (Skyvern → backend, during MFA)

    While the login block waits for a code, Skyvern repeatedly calls a public URL (often exposed through ngrok) with the job id in the query string. The backend looks up the run id, fetches the OTP record, and returns plain text digits. Skyvern may call multiple times; responses must stay consistent with what their client expects until verification completes or times out.

  4. Latest-job helper (optional automation bridge)

    A separate authenticated endpoint can return the current job id and Skyvern run id for workflows that fetch “who is active” before posting OTP. This is convenience, not a substitute for passing ids explicitly when you can.

  5. Manual read API

    For debugging or alternative clients, the backend can expose a JSON read of stored OTP by carrier and run id, with an optional “consume” mode that removes the key after read.

  6. Guards you should understand

    If a poll uses a job id that is no longer the “latest” active job, the backend may reject it so two concurrent quotes do not cross wires. If mapping expired in Redis, polls fail until a new generate-quote refreshes state.

4 n8n — from inbox to backend (sequential story)

n8n’s job is only: turn an email that contains an OTP into a trusted HTTP call that peak-backend can store. It does not drive the browser; Skyvern does.

  1. Step 1 — Something must wake the workflow

    Prefer an email received trigger on the automation mailbox so work runs when a new message actually lands. Polling “get many messages” on a timer works for experiments but is easy to misconfigure (wrong folder, message limits, old mail in the batch).

  2. Step 2 — Confirm it is the right message

    Optional filters (subject, sender, keywords) reduce noise so Travelers OTP mail is not confused with unrelated newsletters.

  3. Step 3 — Extract the six digits

    A function or code step scans subject and body for a six-digit code. If none is found, the branch should stop—there is nothing useful to send yet.

  4. Step 4 — Attach correlation for Redis

    The webhook must tie this OTP to the same Skyvern run the browser is in. In practice that means including the workflow run id returned when the quote was started (and the job id from the backend when you have it). Without that alignment, the backend stores the code under the wrong key and Skyvern’s poll returns empty.

  5. Step 5 — POST to peak-backend

    The HTTP node sends JSON to the OTP webhook path. If the team configured a shared secret, the same secret must appear on the request or the backend refuses the call. A success response means Redis now holds the code for Skyvern’s next poll.

  6. Step 6 — Avoid duplicate posts

    The same email should not fire the webhook twice (dedupe by message id or similar). Duplicate posts can overwrite or confuse timing.

Mailbox reality check: The inbox n8n connects to must be the one where OTP arrives (after forwarding or SMS-to-email). Seeing a message in “Sent” from a personal account does not prove the automation mailbox received it.

5 Skyvern API — how a run actually starts

peak-backend does not open a browser itself. It talks to Skyvern’s JSON API. That call is what creates the workflow run you later see in the Skyvern UI.

  1. Use the API host, not the human UI

    The clickable Skyvern interface and the machine API often live on different ports. The integration must target the API port; otherwise the server answers with HTML and the backend cannot parse a run id.

  2. Identify the workflow

    Each saved workflow has a stable id. The backend sends that id so Skyvern knows which login + navigation script to execute (Travelers portal, credential reference, goals you defined in the UI).

  3. Send parameters for this specific attempt

    When enabled, the payload includes job id, quote id, carrier list, and the merged form data. The integration may also attach a public backend URL and job id so Skyvern’s login block knows where to poll for MFA codes.

  4. Accept the response and learn the run id

    Skyvern returns structured JSON describing the new run. The backend normalises different possible field names into one run id, then continues with Redis mapping and the client response.

  5. Compatibility across Skyvern versions

    Self-hosted installs vary slightly. The integration may try more than one official run endpoint until one succeeds, so upgrades on the Skyvern side are less brittle.

6 Skyvern UI — what the agent does in the browser

The workflow you design in Skyvern is the playbook: open this URL, use this credential set, complete this goal, stop or continue on certain screens.

  1. Credentials live in Skyvern, not in the chat

    The automation browser is isolated. You register Travelers username/password (or a vault-backed credential) once; workflows reference that credential by id.

  2. Login block: username, password, then MFA

    The agent types into fields and clicks sign-in. If Travelers shows SMS verification, the UI may show “send code” and “enter code.” The login block can be configured with a verification URL Skyvern calls to obtain the digits while the page waits.

  3. Parameter injection vs dynamic poll

    You can pass a static placeholder OTP for experiments, but production relies on the poll URL hitting peak-backend, which only returns a code after n8n has pushed it—so the order is: MFA screen appears → Skyvern polls → (parallel) mail arrives → n8n webhook → poll succeeds → agent fills the field.

  4. Why a run can still fail after one successful poll

    Wrong or expired codes, extra “send code” clicks, or Skyvern retrying after Redis no longer has the record all show up as MFA failures. The UI message may still say “failed to get TOTP” even when the underlying issue is verification or timing.

  5. Beyond login — quote steps

    After MFA, additional blocks (or the same workflow) must describe navigating to rating, filling applicant data, and submitting. A workflow that ends immediately after login never completes a quote.

7 Generate-quote path — tying the sequence together

This is the first hop in the automation chain. Everything else (n8n, polling) assumes this hop already created a job id and started Skyvern.

  1. Client submits quote intent

    Typical body includes structured field data for the quote and, ideally, a stable extraction id per document so duplicate submits reuse one in-flight run instead of spawning parallel Skyvern jobs.

  2. Backend chooses Travelers-only behaviour

    Current product focus routes through the Travelers integration path only; other carriers are intentionally not started by this entry point.

  3. Payload shaping for Skyvern

    Sensitive credential references and defaults are merged from server configuration so clients do not have to embed secrets. Optional dev defaults for OTP exist for testing but real runs depend on n8n + Redis.

  4. Skyvern invocation and mapping write

    After the API accepts the run, the backend records job↔run in Redis so OTP storage and poll resolution can use either identifier n8n has available.

  5. Immediate response to the client

    The API answers with “accepted” and returns quote id, job id, and Skyvern run id when present. The browser automation continues asynchronously; the client should not assume the quote is finished when this response returns.

  6. What happens next (by design)

    Then Skyvern runs the browser. Then MFA may appear. Then Skyvern polls peak-backend. In parallel the carrier delivers OTP to mail/SMS. Then n8n posts the webhook. Then polls succeed and the agent can verify and continue.

9 Security & operations

  • Do not commit real environment files or paste API keys into chat; rotate anything that was ever exposed.
  • Treat OTP as sensitive: short TTL, avoid full codes in production logs, mask in operator consoles.
  • Keep Skyvern workflow run id aligned between n8n’s webhook payload and Redis; job id is a bridge when run id is not yet known to n8n.
  • Terminate or cancel old Skyvern runs when starting a new quote to avoid stale polls against outdated job ids.
  • Self-hosted Skyvern often runs beside UI, database, and proxy containers—confirm what is running on your host before debugging network paths.
