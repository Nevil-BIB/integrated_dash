Playwright Flow (Auto-Owners) — peak-backend

This document explains how the current Playwright automation works, which functions run in which order, and how errors are captured and returned back to the frontend.

Current implemented steps: Username → Password → Sign In

1) API entrypoint (start the Playwright job)

Frontend calls this endpoint to start the automation. The backend returns a jobId immediately (HTTP 202), then runs Playwright in background. 

POST /api/generate-quote/playwright

Response (202):
{
  "success": true,
  "message": "Playwright automation job accepted.",
  "data": {
    "jobId": "...",
    "status": "queued",
    "startedAt": "ISO_DATE",
    "step": "secure_sign_in_username"
  }
}

Code: src/controllers/playwright-generate-quote.controller.ts

2) Job status endpoint (frontend polls this)

Frontend can poll job state using the jobId returned from the start call. This is where you will see the final error message and the screenshot path when failures occur. 

GET /api/generate-quote/playwright/:jobId

Response (200):
{
  "success": true,
  "message": "OK",
  "data": {
    "jobId": "...",
    "status": "running|completed|failed",
    "step": "secure_sign_in_username|secure_sign_in_password|...",
    "error": "string (only when failed)",
    "screenshotPath": "playwright-artifacts/....png (only when captured)",
    "startedAt": "ISO_DATE",
    "finishedAt": "ISO_DATE"
  }
}

Code: src/controllers/playwright-generate-quote.controller.ts

3) High-level call flow (who calls what)

Backend call chain

  • Route → POST /api/generate-quote/playwright
  • Controller → generateQuotePlaywrightController() validates env creds and starts job
  • Service → playwrightAutomationService.startJob() creates job in store and runs background task
  • Runner → runAutoOwnersPlaywright(input, { jobId }) launches Chromium, executes steps, detects errors, captures screenshot, throws clean error on failure

Files involved

  • src/routes/quote.routes.ts
  • src/controllers/playwright-generate-quote.controller.ts
  • src/playwright-automation/playwright.service.ts
  • src/playwright-automation/playwright.job-store.ts
  • src/playwright-automation/playwright.runner.ts
  • src/playwright-automation/autoowners.steps.ts

4) Step-by-step (current implemented steps)

These are the steps that currently run when the job starts.

Step 1 — Username screen

  • Navigate to https://www.aoins.com/my.policy
  • Fill username input
  • Click Continue
Function: stepSecureSignInUsername(page, username)
Selectors:
- Username: form#auth_form input[name="username"]#input_1
- Continue: form#auth_form input[type="submit"][value="Continue"]

Code: src/playwright-automation/autoowners.steps.ts

Step 1.5 — Detect AO 500 error after Continue

Sometimes AO returns a maintenance/500 page immediately after clicking Continue. We detect it and fail fast with a screenshot and support ticket id (if available). 

Detection:
- URL contains /errorMaintenance/html/500.html
OR
- Text "Oops, something is broken." visible
Extracts:
- "Support Ticket ID: ..." (if present)
Artifacts:
- playwright-artifacts/<jobId>-ao-500-<timestamp>.png

Code: src/playwright-automation/playwright.runner.ts

Step 2 — Password screen

  • Fill password input (#password)
  • Click Sign In (#submit-button)
Function: stepSecureSignInPassword(page, password)
Selectors:
- Password: form#form input#password[name="password"][type="password"]
- Sign In:  form#form button#submit-button[type="submit"]

Code: src/playwright-automation/autoowners.steps.ts

5) Error handling (what you will see when something fails)

  • Access Denied: detected via URL /vdesk/hangup.php3 OR text Access Denied. (screenshot captured)
  • AO 500 (“Oops, something is broken”): detected via URL/text (screenshot captured + ticket id extracted)
  • Any other error: screenshot captured and error returned

In headed mode (when options.headless=false), the browser will stay open for a short time on error to let you see the failure. 

Option:
options.keepBrowserOpenOnErrorMs
Default: 15000 (15 seconds)

Code: src/playwright-automation/playwright.runner.ts

6) How to run (from frontend)

Start job, store data.jobId, then poll /api/generate-quote/playwright/:jobId until status becomes completed or failed. 

// Start
POST /api/generate-quote/playwright

// Poll
GET /api/generate-quote/playwright/<jobId>

When failed, use data.error and data.screenshotPath to view what happened.

7) Notes / production considerations

  • Job store is currently in-memory (process memory). If the server restarts, job state is lost. For production scale, move to Redis/DB.
  • Browser runs inside the backend process. For concurrency and reliability, move the runner to a worker/queue.
  • Current steps cover login username + password only. Next screens (2FA, navigation, forms) will be added step by step using your HTML selectors.
